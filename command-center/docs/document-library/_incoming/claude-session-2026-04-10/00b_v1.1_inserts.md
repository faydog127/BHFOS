### 3.6 Office System Authority Principle

TIS may capture field data and initiate field actions, but BHFOS remains the authority for record permanence, pricing truth, final document generation, final history, and final audit continuity.

**Authority-Class Split:**

Not all data fields carry the same authority risk. Field/office conflicts are resolved by class:

**Class A — Operational Convenience Fields**
Examples: contact phone, visit notes, scheduling preferences, access instructions, technician prep notes.

- Field may hold session-local precedence for the active session lifecycle.
- BHFOS master record is not overwritten until reconciliation is explicitly approved.
- Session-local values do not automatically become master record truth.
- Reconciliation must occur before archive. (See Section 6.5b for reconciliation clock.)

**Class B — Authorization and Compliance Status Fields**
Examples: service authorization status, contract authority confirmation, compliance flags, payment holds, safety restrictions.

- Office wins absolutely. Field may not override.
- Class B fields are read-only in TIS.
- TIS may display Class B field values. TIS may not write to them.
- Any TIS attempt to write a Class B field generates an immediate error and an audit event.

**Hard-Stop Status Class (a strict subset of Class B):**

The following statuses are hard stops. When any hard-stop status is active on a Customer or Property record, TIS must block any contracting action that depends on valid authorization:

| Status Flag | Blocks |
|-------------|--------|
| `do_not_service` | All field service initiation |
| `no_contracting_authority` | Estimate presentation, work authorization |
| `payment_legal_hold` | Work authorization, certificate issuance |
| `unsafe_entry_restriction` | Session open, visit start |
| `compliance_restriction` | Document issuance, certificate issuance |

Rules:
- A hard-stop status may only be cleared by an authorized BHFOS office user.
- TIS may not clear, override, or bypass a hard-stop status under any condition.
- If a hard-stop status activates during an active session, it is pushed to TIS on next sync. TIS must surface it to the technician immediately. Contracting actions are blocked from that sync point forward.
- Work already formally authorized before the hard-stop activated is not retroactively voided, but office must receive an immediate notification and must confirm or reverse.

### 6.3b Sweep-to-Context Photo Capture

**Purpose:**
Sweep to Context is a field capture mode that allows technicians to quickly capture environmental and location photos during inspection without associating each photo to a specific finding at time of capture. It exists to protect speed without abandoning evidence discipline.

**Behavior:**
- Technician initiates a sweep action (single tap or mode toggle in TIS).
- Photos captured in sweep mode are stored as `context` evidence class.
- Context photos are retained internally in the session record.
- Context photos are not associated with a specific finding at capture time.
- Context photos are not included in customer-facing document renders unless explicitly promoted.

**Promotion Rule:**
- A context photo may be promoted to `finding` evidence class by an authorized user at any time before the session reaches `archived`.
- Promotion is an auditable event: photo ID, actor, timestamp, finding reference.
- Promotion does not back-date the finding. The finding must already exist or be created at time of promotion.
- Promotion is the only path from `context` evidence to customer-visible evidence.

**Evidence Minimum Enforcement:**
- Context photos do not satisfy required finding evidence minimums per `09_Media_Capture_and_Evidence_Standard`.
- A finding with only `context`-class evidence does not meet minimum evidence thresholds.
- A finding must have at least one `finding`-class evidence item (directly associated or promoted) to pass QA.
- System must distinguish `context` vs. `finding` evidence class in all QA checks, evidence counts, and evidence reports.

**Customer-Facing Rendering Rule:**
- All `11x` output documents, the Photo-Evidence Report, and the IAQ Report render only `finding`-class evidence.
- `context`-class evidence is never rendered to a customer-facing artifact unless first promoted.

**Governance Rule:**
- Sweep to Context is not a substitute for proper finding association. It is a speed optimization for site orientation photos.
- If a technician uses sweep mode exclusively without subsequently promoting photos to findings, this is a QA deficiency.
- Repeated excessive `context` photo usage without promotion is a QA-reportable behavior pattern.
- TIS must make the promotion workflow accessible and low-friction to prevent sweep from becoming a dumping ground.

**Audit Triggers:**
- sweep mode initiated
- context photo captured
- context photo promoted to finding evidence (with finding reference)
- finding closed at QA gate with zero promoted evidence (QA flag)

## INSERT 3 OF 6 — New Section 6.5b: Session-Local Precedence and Reconciliation Clock

**Location:** Insert immediately after Section 6.5 (Sync and Office-Grade Record Formation).
**Reason:** Section 6.5 handles successful sync but does not define what happens to session-local Class A values, how long they persist, and what resolves the conflict before archive.

---

### 6.5b Session-Local Precedence and Reconciliation Clock

**For Class A (Operational Convenience) fields:**

Session-local overrides of Class A fields follow this lifecycle:
1. Session-local value is held locally for the active session lifecycle only.
2. It does not overwrite the BHFOS master record automatically.
3. Before archive, a reconciliation gate must resolve all open Class A conflicts.
4. Reconciliation options for each conflict:
   - **Accept field value:** session-local value becomes the new master record value.
   - **Reject field value:** master record value is retained, session-local override is discarded.
   - **Defer with reason:** conflict is flagged as unresolved, a reason is recorded, and reconciliation is deferred to a defined future point.
5. Session-local values that reach archive without resolution and without an explicit deferral are treated as discarded. They do not become master truth by default.
6. The master record is not updated until reconciliation is approved by an authorized user.
7. All reconciliation actions are auditable: field, session-local value, master value, resolution, actor, timestamp.

**For Class B (Authorization/Compliance) fields:**
- No session-local value is ever created.
- TIS is read-only. Any write attempt generates an error and an audit event.

**For Hard-Stop statuses:**
- If a hard-stop status is pushed during an active session:
  - TIS surfaces the hard-stop to the technician on next sync.
  - Contracting actions are blocked from that sync point forward.
  - The session may continue for non-contracting activity (e.g., evidence capture, notes) at office discretion.
  - Office receives notification. Office must confirm or reverse the operational impact.
  - The hard-stop push and technician notification are both auditable events.

## INSERT 4 OF 6 — New Section 6.9b: Tier 2 Correction Boundaries

**Location:** Insert immediately after Section 6.9 (Reopen / Revision Behavior).
**Reason:** The 3-tier correction model's Tier 2 category ("location / asset reference correction") needs hard criteria. Without it, Tier 2 will sprawl into commercial and authorization territory.

---

### 6.9b Tier 2 Correction Boundaries

Tier 2 corrections may change artifact content. They may not cross into commercial, authorization, or classification territory.

**Permitted in Tier 2:**
- Correcting a location label (e.g., `"attic"` → `"second-floor attic access panel"`)
- Correcting an asset reference where the correction does not change which system, zone, or unit bears the finding
- Fixing a typographic error in a finding description that does not change the finding's type, classification, or severity
- Updating a photo label where the underlying evidence content is unchanged

**Not Permitted in Tier 2 — These Require Tier 3 or Full Revision Workflow:**
- Any change to commercial terms: price, scope line, tier, discount
- Any change to the authorization basis: what the customer authorized or was presented
- Remapping a finding across distinct assets (e.g., moving a finding from Unit A to Unit B). This is a finding reassignment, not a location correction. It requires successor artifact generation and an explicit audit reason.
- Any change to severity classification
- Any change that alters customer-facing recommendation framing or urgency

**Boundary Enforcement Rule:**
- The system must prompt the user to classify their correction type at initiation.
- A Tier 2 correction that contains any prohibited change must be rejected at the classification gate.
- The user must re-submit as Tier 3 or full revision.
- Tier classification is an auditable event: tier claimed, tier validated/rejected, actor, timestamp.

**Asset and Zone Reference Rule:**
The Asset/Zone reference layer exists to prevent location label drift. Corrections that stay within the same asset and zone are Tier 2. Corrections that move a finding across asset or zone boundaries are Tier 3 minimum, because they change which physical entity bears the finding — and that changes the customer-facing narrative, the recommendation basis, and potentially the pricing scope.

## 9b. Artifact Revocation and Tombstoning Rules

### 9b.1 Artifact Class Definitions

All Document Artifacts are classified by their revocation obligation:

**Class I — Informational Artifacts**
Examples: Clear Air Checklist, Photo-Evidence Report, IAQ Report (pre-decision stage).

- May be superseded and flagged with a `superseded` status marker.
- Superseded version is retained and historically accessible.
- No distribution enforcement required beyond the status flag.

**Class II — Decision-Bearing Artifacts**
Examples: Estimate / Proposal (once presented to customer), Work Scope (once accepted).

- Must be tombstoned when superseded.
- Tombstoning means: artifact is marked as inactive, non-presentable, and non-actionable.
- Any prior delivery channel (link, PDF, app view) for a tombstoned artifact must render a tombstoned-state page.
- The tombstoned-state page must: display superseded status, identify the current valid successor artifact (if one exists), and require authorized re-issuance if customer action is needed again.
- Session audit trail must record: tombstone event, reason, superseding artifact reference, actor, timestamp.

**Class III — Authorization-Bearing and Signature-Bearing Artifacts**
Examples: Work Authorization, Clean Air Certificate, accepted Estimate version (post-customer decision).

- Must be immediately non-executable when superseded.
- No further customer action (signing, acceptance, authorization) may be taken on a superseded Class III artifact.
- All delivery channels must enforce non-executability — this is not a UI suggestion, it is a system constraint.
- Customer acceptance or signature must be cryptographically or logically bound to the specific active artifact version ID at time of execution.
- A superseded Class III artifact may never be re-activated. Only a new artifact with a new issuance event may replace it.
- The prior artifact remains in the audit record with `superseded` / `inactive` status.

---

### 9b.2 General Tombstoning Rules

Regardless of artifact class:
- No artifact is deleted. All superseded artifacts are retained.
- The current valid artifact for any artifact type is always determinable from session history.
- Tombstoning and successor creation are distinct steps. **Do not tombstone before the successor is ready.** A tombstoned artifact with no valid successor leaves the customer without an actionable document.
- Re-issuing to a customer after tombstoning generates its own audit event: artifact re-issued, reason, actor, timestamp.

---

### 9b.3 Tombstone Trigger Events

A tombstone event is triggered by:
- Revision workflow completion where a new version is rendered for the same artifact class within the same session
- Session reopen where material source change has already affected a presented or issued artifact
- QA rejection that invalidates a customer-issued artifact
- Office-initiated manual supersession with required reason and actor confirmation

---

### 9b.4 Downstream Impact of Tombstone Events

When a Class II or Class III artifact is tombstoned:
- All artifacts that depend on it must be assessed for continued validity.
- The session record must show — at any point in the lifecycle — which artifacts are currently active and which are superseded.

| Tombstoned Artifact | Downstream Artifacts to Assess |
|---------------------|-------------------------------|
| Estimate / Proposal | Work Scope (references accepted option), Work Authorization |
| Work Scope | Estimate (if scope change triggered the revision), Work Authorization |
| Work Authorization | Certificate Eligibility (authorized scope may have changed) |
| Certificate | Session archive readiness (certificate state affects archival gate) |
