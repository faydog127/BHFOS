# TIS UX Inefficiency Scan (2026-04-10)

Scope: Static scan of the TIS field app codebase at `C:\BHFOS\TIS` for interaction-layer inefficiencies (scroll friction, redundant entry, and scouting → proposal carryover).

## What I Scanned

- `C:\BHFOS\TIS\src\pages\AssessmentForm.jsx`
- `C:\BHFOS\TIS\src\pages\EstimateProposal.jsx`
- `C:\BHFOS\TIS\src\utils\speedModeResolver.js`

## Findings (Operator-Friction)

### 1) Long-scroll primary surface (confirmed)

- `AssessmentForm.jsx` is a single stacked form: ~1743 lines, ~12 major `.section` blocks.
- It relies on vertical scrolling to reach later sections; only the action buttons are sticky (`.sticky-actions`), not navigation.

Impact:
- Mobile execution becomes “scroll to complete” instead of “tap to navigate”.
- Context loss increases and completion time increases.

### 2) Redundant “Speed Mode” inputs vs. scouting capture (confirmed)

- `EstimateProposal.jsx` introduces a second input rail (“Speed Mode”) with 5 required inputs:
  - `property_type`, `scope_band`, `access`, `confidence`, `visible_condition`.
- These overlap strongly with scouting capture already present in `AssessmentForm.jsx` / property data.
- The code already *tries* to seed them via `inferSpeedModeInput(...)`, but the defaulting behavior can still force manual correction (example: when count is unknown).

High-leverage root cause:
- `toScopeBand(0)` currently returns `"11-25"` (not “unknown” / blank), which is a “wrong default” that looks like work already done but often isn’t.

### 3) Scouting → proposal is partially “carryover”, partially “parallel system” (confirmed)

What carries:
- Estimate flow reads from the saved `assessment` (e.g., contact placeholders, hazard score inputs) and photos.

What does *not* reliably carry without re-confirmation:
- The 5 fixed Speed Mode inputs (even when seeded, the UI still presents them as an active re-entry rail).

## Recommendations (Surgical, Interaction-Layer)

### A) Convert `AssessmentForm` from “scroll UI” → “step UI”

Add a sticky bottom navigation with 4–6 steps (tap targets), e.g.:
- Capture
- Findings / Signals
- Access
- Contact
- Notes
- Summary

Each step renders a compact panel (progressive disclosure); no “complete the task by scrolling”.

### B) Make Speed Mode a confirmation surface, not a second form

Target behavior:
- Auto-seed from `assessment/property/photos` (already exists).
- Only require taps for *missing* or *uncertain* inputs (blank state), not for prefilled values.
- Remove “wrong defaults” that create rework (start with `scope_band`).

### C) Explicit “Promotion” moment when moving from scouting → proposal

On “Estimate & Proposal” transition:
- “Promote scouting inputs to Speed Mode defaults” (one tap), then open proposal.
- Keep an explicit “override” affordance for the 5 inputs, but don’t force re-confirmation on every visit.
